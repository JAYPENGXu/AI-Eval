# AIAssistant 一期功能说明

这份文档面向第一次接触项目的人，说明一期 RAG 系统到底做了什么、页面怎么用、接口怎么协作、数据库保存了什么，以及后续如何维护它。

## 1. 一期目标

一期的目标不是先做复杂 Agent，而是先做一个可靠、可调试、可评测的 RAG 知识库问答系统。

用户可以上传知识文档，系统把文档切成 chunk，建立向量索引和关键词索引。之后用户在右侧对话栏提问，系统会检索相关 chunk、重排、压缩上下文，再调用大模型生成流式回答。

同时，系统会把每次问答的完整过程保存下来，让我们知道：

- 问题有没有被正确改写。
- Vector Search 有没有召回正确 chunk。
- BM25 有没有召回正确 chunk。
- Hybrid Fusion 有没有把正确 chunk 保留下来。
- Rerank 有没有把正确 chunk 排到前面。
- Context Compression 有没有丢掉关键句。
- 最终大模型有没有答对。

这就是一期最重要的价值：不只回答问题，还能解释为什么答对或答错，并把失败沉淀成回归评测集。

## 2. 用户能做什么

当前系统已经具备这些能力：

- 注册、登录、进入自己的 RAG 工作台。
- 创建和使用知识库。
- 上传知识文档。
- 选择不同切片方式并预览切片效果。
- 一键索引文档，生成 chunk 和 embedding。
- 在对话栏中基于知识库问答。
- 使用 SSE 看到流式回答。
- 调整 RAG 参数并观察每一层检索结果变化。
- 查看每次问答的 Debug Trace。
- 对比两次 Trace。
- 维护评测基准问题。
- 运行 RAGAS 评测。
- 查看评测报告、检索指标、失败原因。
- 从 Trace 或评测失败中一键沉淀 Regression Case。
- 一键重置工作区，清理 SQLite 业务数据和向量库 chunk。

## 3. 页面布局

前端主页面分成三列。

左侧负责知识库和文档：当前知识库、上传文档、选择切片方式、预览 chunk、索引、重置工作区。

中间是“调试工作台”，包含四个选项卡：

- `Debug`：切片实验室、RAG 检索调试、参数面板。
- `Evaluation`：运行评测、查看评测报告、Run 对比、Failure Analysis。
- `Datasets`：维护评测集、基准问题、回归集。
- `History`：查看历史 Trace、Trace 详情、Trace 对比、一键转 Regression。

右侧是 RAG 对话栏：用户输入问题，前端发起 SSE 请求，后端边生成边返回 token，回答完成后保存 ChatMessage 和 RagTrace。

## 4. 一次问答的完整流程

```text
用户原始问题
-> Query Rewrite
-> Vector Search
-> BM25 Search
-> Hybrid Fusion
-> Rerank
-> Context Compression
-> Final Prompt
-> LLM 生成回答
-> SSE 返回前端
-> 保存 RagTrace
```

Query Rewrite 支持：

- `none`：不改写。
- `rule`：默认策略，使用规则提取关键词。
- `llm`：调用大模型改写。

检索使用改写后的查询，但最终回答仍然回答用户的原始问题。

## 5. 切片能力

系统支持五种切片方式：

- Token 切片。
- 句子切片。
- 句子窗口切片。
- 语义切片。
- Markdown 切片。

默认是句子切片。前端的“切片实验室”可以直观展示每种切片方法的差异。用户调整 `chunk_size` 后，可以立刻看到 chunk 数量、chunk 内容和边界变化。

后端索引时会把 chunk 写入 SQLite 的 `Chunk` 表，并为 chunk 生成 embedding。向量索引写入 Milvus，业务元数据保留在 SQLite 中。

## 6. 检索与生成能力

Vector Search 使用 Embedding 表示语义相似度，适合召回语义相近但字面不完全一致的内容。

BM25 是关键词检索，适合精确词、标题、专有名词、工具名、字段名。

Hybrid Fusion 使用 RRF 把 Vector 和 BM25 的候选结果合并，避免只依赖一种检索方式。

Rerank 会对 Hybrid Top N 候选重新排序。前端会展示 Rerank 前后顺序变化，帮助判断 reranker 是否把正确 chunk 提前。

Context Compression 会把 Rerank 后的 chunk 再压缩，只保留与问题最相关的句子。前端展示压缩前文本、压缩后文本、token 节省比例和关键句是否保留。

## 7. 调试参数

调试工作台可以调这些参数：

- `chunk_size`：切片大小。
- `top_k`：最终检索返回数量。
- `BM25 top_k`：BM25 候选数量。
- `RRF_K`：RRF 融合参数。
- `Rerank top_n`：重排后保留数量。
- `Compression 策略`：上下文压缩方式。
- `Query Rewrite 策略`：问题改写方式。

每次参数变化后，用户都可以重新问答或运行评测，观察各阶段结果怎么变化。

## 8. Trace 历史记录

每次 RAG 问答都会保存 `RagTrace`。

Trace 记录包括：

- 原始问题。
- 改写后的查询。
- 改写策略。
- Vector Search 结果。
- BM25 Search 结果。
- Hybrid Fusion 结果。
- Rerank 前后结果。
- Compression 前后上下文。
- Final Prompt。
- 最终回答。
- 使用的参数。

在 `History` 选项卡可以查看 Trace 列表、Trace 详情、选择两条 Trace 对比，也可以从 Trace 一键生成 Regression Case。

## 9. 评测集管理

一期已经把“评测基准表”升级成“评测集管理”。

每个评测 Case 包含：

- `question`：问题。
- `reference`：标准答案。
- `expected_terms`：回答里应该包含的关键词。
- `target_chunk_ids`：标准答案所在 chunk。
- `suite`：评测集，例如 `smoke`、`benchmark`、`regression`、`release`。
- `tags`：标签。
- `difficulty`：难度。
- `source`：来源。
- `notes`：维护备注。
- `enabled`：是否启用。

推荐这样维护：

- `smoke`：少量快速冒烟用例，用来确认系统基本可用。
- `benchmark`：专家维护的核心基准集，用来衡量系统真实能力。
- `regression`：历史失败沉淀出来的回归集，用来防止老问题复发。
- `release`：上线前必须通过的关键用例集合。

如果数据库没有启用的 Case，后端会 fallback 到默认 JSON 示例，保证新项目第一次运行也能看到评测效果。

## 10. 评测报告

评测可以从前端直接运行。前端点击“运行评测”后，后端会启动轻量后台线程，前端拿到 Run ID 后每 2 秒轮询一次状态。

评测结果保存到 SQLite：

- `RagEvalRun`：一次评测运行。
- `RagEvalCaseResult`：每个 Case 的详细结果。

评测报告展示：

- RAGAS 分数。
- Hit Rate。
- Recall@K。
- MRR。
- Vector 是否命中目标 chunk。
- BM25 是否命中目标 chunk。
- Hybrid 是否命中目标 chunk。
- Rerank 后是否仍保留目标 chunk。
- Compression 是否保留关键句。
- 最终回答是否正确。

还支持选择两个 Run 做对比，观察不同参数组合对结果的影响。

## 11. Failure Analysis

Failure Analysis 面板会把失败 Case 按原因分组。

失败原因包括：

- `rewrite_failed`：问题改写失败。
- `vector_miss`：向量检索没有命中目标 chunk。
- `bm25_miss`：BM25 没有命中目标 chunk。
- `hybrid_drop`：单路检索命中了，但融合后丢失。
- `rerank_drop`：Hybrid 命中了，但 Rerank 后没有保留。
- `compression_lost`：Rerank 命中了，但压缩后丢掉关键句。
- `llm_answer_wrong`：上下文足够，但最终回答不正确。
- `no_reference`：没有参考答案，无法判断。
- `target_chunk_stale`：目标 chunk 已过期或不存在。

这些原因来自链路中每个阶段保存的诊断字段，而不是只根据最终回答主观判断。

## 12. 从失败到回归集

系统支持两个入口把失败沉淀成 Regression Case：

- 从 History 的 Trace 详情点击 `To Regression`。
- 从 Evaluation 的失败 Case 点击 `To Regression`。

生成后，这个 Case 会进入 `regression` suite。以后每次改切片、检索、重排、压缩、Prompt 或模型，都可以跑 regression，确认历史失败不会重新出现。

这是当前系统最重要的闭环能力。

## 13. 前后端如何交互

注册登录：

```text
Vue 登录页
-> POST /api/auth/login/
-> Django 校验用户
-> 返回 JWT
-> 前端保存 token
```

上传和索引文档：

```text
Vue 上传文档
-> POST /api/documents/
-> Django 保存 Document
-> POST /api/documents/{id}/chunk-preview/
-> POST /api/documents/{id}/index/
-> Django 生成 Chunk
-> 调用 Embedding API
-> 写入 SQLite + Milvus
```

RAG 问答：

```text
Vue 发送问题
-> SSE 请求后端 chat endpoint
-> Django 执行 RAG Pipeline
-> LLM 流式生成
-> SSE delta 返回前端
-> 保存 ChatMessage 和 RagTrace
```

运行评测：

```text
Vue 点击运行评测
-> POST /api/rag-eval-runs/run/
-> Django 创建 RagEvalRun
-> 后台线程执行评测
-> Vue 轮询 Run 状态
-> 展示 RagEvalCaseResult
```

失败沉淀：

```text
Vue 点击 To Regression
-> POST /api/rag-benchmark-cases/from-trace/
或 POST /api/rag-benchmark-cases/from-eval-case/
-> Django 创建 RagBenchmarkCase
-> suite = regression
```

## 14. 数据库保存什么

SQLite 保存业务数据和调试评测数据：

- 用户、认证相关表。
- `KnowledgeBase`：知识库。
- `Document`：上传文档。
- `Chunk`：切片文本、文档来源、chunk 元数据。
- `ChatSession`：会话。
- `ChatMessage`：聊天消息。
- `RagTrace`：一次问答的完整链路记录。
- `RagBenchmarkCase`：评测基准和回归用例。
- `RagEvalRun`：一次评测运行。
- `RagEvalCaseResult`：单个 Case 的评测结果。

Milvus 保存向量索引，用于 Vector Search。SQLite 中的 Chunk 是业务主数据，Milvus 中的向量可以重建。

## 17. 当前能力评价

当前一期系统已经不是一个简单 Demo，而是一个具备工程闭环的 RAG 调试与评测系统。

它的核心优势是：

- 链路透明：每一层结果都能看到。
- 参数可调：每个关键变量都能实验。
- 结果可评：不是只靠肉眼看回答。
- 失败可沉淀：真实失败能变成 regression。
- 长期可演进：以后接 Agent 时，有可靠的 RAG 和 Eval 底座。

下一阶段可以继续加强：

- 更严格的权限和多用户隔离。
- 更完整的异步任务队列。
- 更强的文档解析能力。
- 更细的成本、延迟、模型调用监控。
- Agent 工具调用和任务评测。
