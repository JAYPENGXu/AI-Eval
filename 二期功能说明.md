# AIAssistant 二期功能说明：RAGOps Agent、LangGraph 与 HITL 工作流

二期目标不是给系统加几个“炫技按钮”，而是在一期 RAG 调试和评测底座上，做一个实用的 RAGOps Agent 工作流。

当前 Agent 已经收敛成一个主入口：

> 端到端 RAG 修复工作流

它围绕失败 Trace 或 Baseline Eval Run 自动收集证据、定位失败阶段、生成优化方案，并在真正执行写操作前让人类确认。

## 1. 为什么需要 Agent

一期已经能看到很多数据：

- Trace。
- 检索结果。
- Rerank 结果。
- Compression 结果。
- Final Prompt。
- Eval Run。
- Failure Analysis。
- 成本数据。

但这些数据多了以后，用户仍然要自己判断：

- 问题到底出在哪一层？
- 是 Query Rewrite 偏了，还是 BM25 没召回？
- 是 Rerank 掉了正确 chunk，还是 Compression 丢了关键句？
- 下一步应该调什么参数？
- 这次失败要不要沉淀成 Regression Case？
- 参数实验要不要跑？

Agent 的价值就是把这些诊断步骤编排成一个工作流，而不是让用户在多个页面之间来回找线索。

## 2. 当前 Agent 工作流

```text
用户选择失败 Trace 或 Baseline Eval Run
↓
Agent Planner 规划需要读取哪些证据
↓
Tool Executor 调用只读工具
↓
Diagnosis 判断失败阶段
↓
Proposal 生成优化建议和候选动作
↓
Human Decision 生成待确认 Action Card
↓
用户确认后执行写操作
↓
审计记录保存执行结果
```

当前前端只保留一个主工作流入口，不再保留四个松散任务卡片。

## 3. LangGraph 节点设计

后端 Agent 位于：

```text
backend/rag/agent/
├── graph.py
├── tools.py
├── checkpointing.py
└── services.py
```

LangGraph 节点：

- `planner`：根据用户目标和业务上下文决定要读取 Trace、Eval Run 还是成本摘要。
- `tool_executor`：执行只读工具。
- `diagnostician`：基于工具结果判断失败阶段。
- `proposal`：生成建议动作，例如创建 Regression Case 或运行参数实验。
- `human_decision`：把需要写操作的建议转成 Action Card。
- `responder`：生成最终诊断报告。

## 4. Agent 工具

当前工具以只读诊断为主：

- `get_trace_detail`：读取单条 Trace 的问题、改写、检索、重排、压缩、答案和 Prompt 摘要。
- `compare_eval_runs`：比较两个 Eval Run 的指标、参数和 case 变化。
- `get_model_usage_summary`：读取模型调用次数、token、成本、慢请求和失败请求。

这些工具返回的是摘要和 ID，不把完整大文档、大 chunk、大 Trace 塞进 Agent state。

## 5. Human-in-the-loop

二期的重点是：Agent 可以建议，但不能擅自执行写操作。

写操作包括：

- 从 Trace 创建 Regression Case。
- 从 Eval Failure 创建 Regression Case。
- 从用户负反馈创建 Regression Case。
- 运行参数实验计划。

这些动作都会先创建 `RagAgentAction`：

- `pending`：等待用户确认。
- `completed`：已执行成功。
- `failed`：执行失败。
- `rejected`：用户拒绝。

前端展示“待确认动作”，用户点击确认后才调用后端执行。

## 6. Agent 审计

`RagAgentAction` 保存：

- 动作类型。
- 来源：Trace、Eval Failure、User Feedback、Experiment Plan。
- 标题和描述。
- confirm label。
- payload。
- 状态。
- 执行结果。
- 错误信息。
- 创建时间、确认时间、完成时间。

这让 Agent 的每次建议和执行都有审计记录，避免黑盒自动化。

## 7. LangGraph Checkpointer

项目使用独立 SQLite 文件保存 LangGraph checkpoint：

```text
backend/agent_state/langgraph_checkpoints.sqlite3
```

工程约束：

- 不混用 Django `db.sqlite3`。
- Django 表保存业务事实。
- Checkpointer 保存 Agent 执行状态。
- `thread_id` 与业务上下文绑定，便于 resume 和审计。
- Graph state 不保存大文档、大 chunk 或完整 Trace，只保存 ID 和必要摘要。

相关配置：

```text
LANGGRAPH_CHECKPOINT_DB=...
```

默认路径来自 `assistant_backend/settings.py`。

## 8. Thread ID 设计

前端基于当前业务上下文保存 Agent thread id：

- 知识库。
- Trace。
- Eval Run。
- Compare Eval Run。

后端调用 LangGraph 时传入：

```python
config = {"configurable": {"thread_id": thread_id}}
graph.invoke(initial_state, config=config)
```

当前已经保存状态，但还没有实现完整 interrupt/resume。下一步可以把 `human_decision_node` 改成真正的 LangGraph interrupt。

## 9. 参数实验工作流

Agent 可以基于 Baseline Eval Run 的失败 case 生成实验计划：

```text
读取 Baseline 失败 case
↓
判断主要失败阶段
↓
生成 2-3 套参数实验方案
↓
创建待确认 Action Card
↓
用户确认
↓
批量创建 Eval Run
↓
刷新实验结果
↓
推荐 Winner
```

相关模型：

- `RagExperimentPlan`
- `RagExperimentVariant`
- `RagEvalRun`

当前不会自动应用默认参数，只推荐 winner。

## 10. 前端 Agent 页面

当前 Agent 页面已经做减法，只保留：

- 工作流说明。
- 四步流程展示。
- 失败 Trace 选择。
- Baseline Eval Run 选择。
- 可选 Compare Eval Run。
- 优化目标输入。
- 启动前二次确认。
- 诊断报告。
- 参数实验计划。
- 人工确认动作。
- 执行计划。
- 工具调用。
- 历史审计记录。

这比“多个 Agent 按钮”更适合演示 LangGraph 和 HITL 的真实价值。

## 11. 与一期能力的关系

二期 Agent 不替代一期 RAG 系统，而是消费一期产生的数据。现在一期还提供了 Query Router 和 Session Summary Memory，Agent 可以把它们作为诊断证据：

- 读取 `RagTrace` 做单次失败诊断。
- 读取 `RagEvalRun` 和 `RagEvalCaseResult` 做批量失败分析。
- 写入 `RagAgentAction` 做 HITL。
- 写入 `RagBenchmarkCase` 做回归沉淀。
- 读取 `query_intent`、`route_decision` 判断问题是否被正确路由。
- 读取 `ChatSessionSummary` 和 Trace 中的 `session_summary_*` 字段判断多轮记忆是否参与改写。
- 创建新的 `RagEvalRun` 做参数实验。

所以一期是 RAG 工程底座，二期是 RAGOps 自动化工作流。

## 12. 当前限制

当前二期仍是 MVP：

- Checkpointer 已接入，但 Human Decision 还不是 LangGraph 原生 interrupt。
- Agent 工具数量有限，主要用于 RAG 诊断。
- 文档解析/索引、Agent 参数实验、Eval Run、解析评测和 Session Summary 已统一迁移到 Celery + Redis，并保存任务运行状态。
- Agent 记忆目前是 checkpoint 线程状态；RAG 对话已有会话摘要记忆，但还不是跨会话用户长期画像。
- Agent 对失败阶段的判断仍依赖已有 Trace / Eval 字段质量。

## 13. 下一步建议

优先级较高：

- 把 `human_decision_node` 改成真正 interrupt/resume。
- 增加 Trace Replay / 参数重跑。
- 增加 Agent 工作流级评测：任务完成率、工具调用正确率、HITL 转化率。
- 增加 Agent 输出结构化 schema，减少自然语言报告漂移。
- 引入更稳定的后台任务队列。

中长期：

- 支持工具注册中心。
- 支持更多 Function Calling 工具。
- 把 RAG 会话摘要作为 Agent 诊断证据，判断问题改写是否因记忆不足失败。
- 支持跨 session 的长期偏好记忆。
- 支持团队协作式多 Agent，但前提是单 Agent 工作流足够稳定。

## 14. 二期总结

二期当前完成的是：

- LangGraph 工作流骨架。
- SQLite checkpointer。
- thread_id 绑定业务上下文。
- HITL Action Card。
- Agent 审计记录。
- RAG 修复工作流入口。
- 参数实验计划与 winner 推荐。

它的定位是“RAGOps 诊断与修复 Agent”，不是泛用聊天 Agent。


## 14. 参数实验、Release Gate 与配置回滚

Agent 参数实验采用两阶段 HITL：第一次确认启动受控 Variant；Winner 必须满足评分提升、失败数不增加、Deterministic/Judge 通过和性能约束。合格 Winner 保存为不可变 Candidate，并自动运行 `release` suite。

Release 通过后才创建 `publish_rag_config` Action Card。第二次人工确认原子切换知识库的 `active_config_version`。历史已部署版本可通过独立 `rollback_rag_config` 卡回滚，所有操作写入 `RagConfigDeployment`。
